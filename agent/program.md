# 小红书文案自主优化框架

> 借鉴 autoresearch 的核心循环：定义标准 -> 生成 -> 打分 -> 找差距 -> 改 -> 重复。
> 目标不是一次写完，而是让 AI 在约束下持续迭代出更像人写、也更像你的文案。

## 文件依赖

| 文件 | 作用 | 是否必须 |
|------|------|---------|
| `program.md` | 核心逻辑：生成、评分、迭代、存档、学习 | 必须 |
| `style-profile.md` | 持久化风格规则 | 必须 |
| `memory-bank.md` | 真实终稿记忆库，保存用户最终选用或人工修改后的文案样本 | 强烈建议 |
| `bootstrap.md` | 不支持读文件时的一站式启动 prompt | 可选 |
| `history/` | 保存任务过程、迭代稿、最终稿、学习样本 | 可选但强烈建议 |

## Agent 首次加载指令

1. 读取本文件 `program.md`
2. 读取 `style-profile.md`
3. 如果存在，再读取 `memory-bank.md`
4. 如果 `style-profile.md` 或 `memory-bank.md` 不存在或为空，按默认规则生成，不报错
5. 如果能写文件，默认把任务过程存到 `history/`

---

## 一、评分标准（满分 50 分）

每个维度 1-10 分。所有优化都围绕这 5 个维度展开。

| 维度 | 10 分标准 | 常见扣分点 |
|------|-----------|-----------|
| 钩子力 | 标题和开头让人停下来 | 平、散、没有情绪或信息差 |
| 共鸣感 | 读者会觉得“这说的就是我” | 场景太泛、像在讲大道理 |
| 信息密度 | 每句话都有增量 | 废话、重复、空洞形容词 |
| 行动力 | 读完想收藏、想试、想评论 | 没有方法、没有带走点 |
| 平台适配 | 很像小红书自然内容 | 书面语重、排版差、搜索词弱 |

### 终止条件

- 总分 >= 40/50
- 无单项 < 7
- 硬性约束全部满足
- 连续 2 轮无提升时，取历史最佳版本停止

## 二、输入解析（允许自由表达）

用户可以用两种方式开始：

### 方式 A：结构化输入

```text
主题：___
目标人群：___
内容类型：___
参考调性：___
关键词：___
```

### 方式 B：自由表达

```text
我想写一篇介绍某个工具的种草文。
读者是经常发小红书、想提高文案质量的人。
希望更像爆款口吻，不要太像 AI。
```

### 处理规则

1. 先从用户原话里推断：主题、目标人群、内容类型、调性、关键词、附加要求
2. 缺什么补什么，不要一上来强制用户重新按模板填写
3. 只有在无法安全推断、且会明显影响文案质量时，才追问少量关键问题
4. 用户后续新增的要求，全部进入当前任务约束，不视为临时备注

---

## 三、硬性约束系统

除了 5 维评分，还要维护一份“当前任务约束清单”。

默认至少包括：
- 推荐主标题必须在 20 字以内
- 文案风格符合小红书阅读习惯
- 遵守 `style-profile.md`
- 如果已有真人终稿记忆，尽量贴近 `memory-bank.md` 中的表达方式

用户可追加的约束示例：
- 更像爆款口吻
- 更科技感
- 少一点 emoji
- 不要太像 AI
- 多一点种草感
- 保留某个表达

### 约束执行规则

1. 每轮生成前先读取当前约束清单
2. 每轮评分后增加“约束检查”
3. 有任何硬性约束不满足，都不能直接定稿
4. 后续迭代必须继续带着这些约束，不允许忘掉用户后来补充的要求

## 四、文案生成模板

### 4.1 标题生成（3 个备选）

标题公式可混用：

```text
数字法：X 个___，第 N 个绝了
对比法：别再___了，试试___
身份法：___人必看，___
悬念法：原来___的关键是___
结果法：用了___后，我终于___
痛点法：你总是___，其实是因为___
```

### 标题要求

- 推荐主标题必须 <= 20 字
- 其他备选标题默认也尽量 <= 20 字
- 标题之间要有差异，不要 3 个都一个模子
- 尽量包含至少一个强停留元素：数字、反差、痛点、结果、情绪词

### 4.2 正文生成

```markdown
【结构】
- 开头钩子：1-2 句，制造共鸣、反差或悬念
- 主体：3-5 个要点，每个要点都有实际信息增量
- 收尾：1-2 句，做总结并引导互动/收藏

【格式规范】
- 总字数：300-800 字
- 每段不超过 3 行
- emoji：按用户风格要求决定，默认适量
- 语气：口语化、像在和人说话
- 禁止：公文腔、汇报腔、过度营销腔
```

### 4.3 标签生成

- 5-10 个标签
- 前 3 个优先核心搜索词
- 其余补场景词、长尾词、情绪词

## 五、评分格式

每轮都输出评分卡：

```text
## 评分卡 —— Round N

| 维度 | 分数 | 扣分原因 | 改进方向 |
|------|------|---------|---------|
| 钩子力 | ?/10 | 具体问题 | 具体建议 |
| 共鸣感 | ?/10 | 具体问题 | 具体建议 |
| 信息密度 | ?/10 | 具体问题 | 具体建议 |
| 行动力 | ?/10 | 具体问题 | 具体建议 |
| 平台适配 | ?/10 | 具体问题 | 具体建议 |
| **总分** | **??/50** | | |

### 约束检查
- 主标题 <= 20 字：通过 / 不通过
- 用户新增要求：通过 / 不通过
- 风格档案一致性：通过 / 不通过
- 终稿记忆贴合度：通过 / 不通过 / 暂无记忆

最低分维度：___
下一轮优化焦点：___
```

### 评分纪律

- 8 分代表真实有停留和收藏可能，不是“写得没错”
- 每个扣分都要对应到具体句子或结构问题
- 如果约束没满足，不要因为分数够了就强行定稿

## 六、迭代循环

```text
LOOP（最多 5 轮）

1. Round 1 生成完整初稿（标题 × 3 + 正文 + 标签）
2. Round 2+ 只改最低分维度，其他部分尽量不动
3. 每轮都评分，并做约束检查
4. 总分提升则 keep；持平或下降则回退到上一轮最佳版本
5. 连续 2 轮无提升则停止，取历史最佳版本
```

### 记录要求

每轮都记录：

| Round | 总分 | 钩子 | 共鸣 | 密度 | 行动 | 适配 | 状态 | 改了什么 |
|-------|------|------|------|------|------|------|------|---------|

### 保存要求

如果能写文件，在 `history/YYYY-MM-DD-主题/` 下至少保存：

- `brief.md`：推断后的任务信息与约束清单
- `round-01.md`、`round-02.md` ...：每轮完整文案
- `notes.md`：每轮为什么这样改
- `final.md`：当前 AI 定稿
- `selected-final.md`：用户最终选用的版本（如有）
- `human-final.md`：用户人工修改后的终稿（如有）
- `memory-update.md`：本次新增的表达记忆摘要

如果用户后续继续追加要求：
- 从当前最佳版本继续迭代
- 重新打分
- 新一轮稿件继续保存，不覆盖旧轮次

## 七、终稿记忆系统

这是和 `style-profile.md` 并行的一套记忆机制。

### 7.1 区别

- `style-profile.md`：保存抽象规则，例如偏好什么语气、避免什么词
- `memory-bank.md`：保存具体样本记忆，例如用户最终真的会发的文案长什么样

### 7.2 触发条件

当用户明确提供以下内容时触发记忆写入：

- 最终选用版本
- 人工修改后的最终发布版
- 明确说“以后按这个感觉写”的样本

### 7.3 写入动作

1. 保存全文到当前任务目录
2. 在 `memory-bank.md` 中追加索引
3. 提取以下信息：
   - 开头习惯
   - 常用句长
   - 语气强弱
   - 偏好词/避免词
   - 喜欢的转折方式
   - 收尾方式
4. 在 `memory-update.md` 里记录本次新学到的东西

### 7.4 使用原则

- 最新的终稿样本优先级更高
- 人工修改后的版本优先级高于 AI 原稿
- 学习的是表达习惯，不是照抄旧内容
- 如果当前任务和历史样本冲突，优先服从用户当前明确要求

## 九、风格学习系统

当用户提供 AI 定稿和人工最终版时：

1. 逐项对比标题、开头、结构、用词、emoji、收尾
2. 提取稳定偏好和明显禁忌
3. 标注置信度
4. 先给用户确认
5. 用户确认后更新 `style-profile.md`
6. 无论规则是否升级，都把本次终稿样本写入 `memory-bank.md`
7. 把本次样本与 diff 结论保存到 `history/` 对应任务目录

### 学习原则

- 人工修改版优先级最高
- 单次现象先记为“观察”或“偏好”，不要过早升级为铁律
- 风格学习不能覆盖用户在当前会话里明确提出的新要求，二者应叠加执行
